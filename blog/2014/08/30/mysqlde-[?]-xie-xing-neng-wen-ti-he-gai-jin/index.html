MySQL的一些性能问题和改进

Aug 30th, 2014 2:03 pm

@ 楼方鑫 MySQL并发控制&限流设计

一些性能问题和改进

以及一些他做的修改

性能:

褒贬不一:

业务需求不一样
掌握力度不同
flash设备与传统设备
对开发人员的支持力度

测试

实际:读比写强

查询强:
    简单的SQL(PK或者有效的索引查找)
    要求:内存里完成排序,分组统计,无子查询
更新弱:
    写存储引擎日志
    写MySQL binlog
    server&storage层的分布式事物

一些问题:

  • 全局锁太多
  • 并发是关键
  • 读写没分开
  • 优化器太弱

全局锁太多

- SYS CPU太高
    TRX_list
    read View List
    Lock List
- Spinlock & RW Lock & Mutex
- 临界区太大
- 大锁 (事务表 buffer index commit)

并发是关键

降低并发减少SYS CPU
合理的并发时程减少锁争用
减少失误表长度 缩短事务表的临界区
控制表的大小,缩短索引维护的临界区

读写要分开

调整优化应用的读写比例,MySQL无法分别控制,以及配合应用的读写比例

优化器弱:SQL写法

子查询 差
join 差
索引选择 不可靠

改进:

  • 多buffer pool(5.5)
  • 读写事物双队列(5.6)
  • 读事物优化&索引全局锁(5.7)

多Buffer Pool

代码重构 & Plugin API重构
MySQL 5.5的主要改进
不能按指表定Buffer Pool,运气重要
代码中到处可见的kernel_mutex
读写互相影响很重要

读写事物双队列

MySQL 5.6在不见了kernel_mutex。
最重要的是读写事务分了不同的队列。
    ro_trx_list
    rw_trx_list
读写操作并发的情况明显改善。

读事物优化&索引全局锁

查询不需要申请事物ID
    自动识别只读操作
    不再申请事物ID,减轻死锁检测的代价
去掉索引全局锁
    按索引range查询得到明显改善
    索引增大时DML的速度明显改善

并发控制:(这应该是他得特色)
